Blog 11 - Problem Solving

  • Tell your friend about a time you got blocked on a simple problem
    • What was the problem?

      • In one of the first sprints, I was learning how to use Command Line. The task was asking me to create new files, move them around, and delete them, all through Command Line. Things were running relatively smoothly until I tried to move one file that kept producing an error!

    • What problem solving techniques did you use?

      • I changed the way I wrote out the command, moved other files around successfully, went in and out of the folders I was using, and could not figure out what was going on. Eventually, I posted my issue on Slack. It turns out that I had simply misspelled the file name, so I when I was trying to move the correctly-spelled file, it was calling upon something that didn't actually exist.

    • How did you feel throughout the process?

      • Fortunately the issue didn't last for that long, so I grew frustrated but not as frustrated as I have been for other, bigger problems that I have had trouble figuring out.

    • What did you learn?

      • I learned the importance of spelling! Especially when so much of our code is being written out (specifically when we're naming new things), it's easy to mispell words, lose a letter, etc. The first thing I check now when I get an error is if I spelled all my commands accurately!

  • Tell your friend about a time you solved a problem in an elegant way
    • What was the problem?

      • One problem that I solved in an elegant way was one of the exercises from Eloquent JavaScript. It asked you two write a program that listed all numbers from 1-100, but replace numbers that are divisible by 3, 5, and boh 3 and 5 with a silly word instead of the number itself.

    • What problem solving techniques did you use?

      • To solve this problem, I wrote out what needed to happen in plain English. I then replaced my words and sentences with code. For one piece of the solution, I scrolled up where there was a reference made earlier to something similar, and I borrowed that hint.

    • How did you feel throughout the process?

      • I felt good. It was a challenge that was difficult but not overwhelmingly so. I stubled a bit here and there, but overall was able to work my way through the issue with enough time and without pulling out any hair.

    • What did you learn?

      • I learned that I can do it! I learned that I actually learned something - yay! I also learned that writing things out in plain English is a helpful way to start breaking down a coding problem.

  • Reflect on how confident you feel using the problem solving techniques and process:
    • Pseudocode: I learn through doing, so I have found Sprint 5's exercises and putting things into practice really helpful in enhancing my understanding of JS.
    • Trying something: As mentioned above, this is definitely my go-to method of problem solving.
    • Rubber ducky method: I also really like this method of problem solving/enhancing understanding. The degree to which I'm able to explain a concept is an indication of how much I understand it. When I review my notes, I try to rephrase things to see if I can do that. This method is a great way to check in with yourself - a "self-assessment" if you will - to see where you're at.
    • Reading error messages: Not very helpful. I can often receive the same error message even after I make a change, so I don't find it useful.
    • Console.logging: I don't really know yet how this can help with problem solving.
    • Googling: Definintely! That's probably my first resource after my own notes when trying to figure out how to do something.
    • Asking your peers for help: Moderately. I try to solve things on my own first, because I feel like that contributes to my learning the most. I have learned that asking earlier in my process when I'm stuck is better than later, because if I wait too long I become too exasperated to work with the help that is being offered.
    • Asking coaches for help: Ditto as above.
    • Improving your process with reflection: I think the reflections are similar to the rubber duck method in that you reflect upon how much you understand something.
    • Describe what these functions do in your own words:
      • .map() - this function applies a secondary function to every element within an array, and then outputs that result into a new array. For example, you have function double = x * 2 that you want to apply to all the numbers inside an array. First, you define your array and your function (in this case, double). Then, call .map(): var mapped = array.map(double(array)). The output will be a new array that is made up of the old array elements each multiplied by 2.
      • .filter() - this function is as it sounds: it creates a new array containing all the elements of an array that pass through a given filter. Here, you have to define what that filter is first, and then apply .filter to your array.
      • .reduce() - this function runs through an array, comparing each element to the array and reducing that element to a value. That value is accummulated, meaning the value gets added to previously returned values. Often this can look like adding the given value to the sum total of the previous array elements. If the array = [0,1,2,3], when we pass it through the .reduce() function, 0 gets added to 1 to get a sum of 1. Then 2 gets added to the accummulated value (in this case, 1) to return 3. Then 3 gets added to our previous total, 3 to output 6. Each element within the array gets added to the accumulation of previous elements in order to reduce the array elements to one value.
